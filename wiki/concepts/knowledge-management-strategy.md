# 우리의 지식관리 전략

이 페이지는 Andrej Karpathy의 **LLM Wiki** 패턴을 우리 위키(`~/wiki`)에 어떻게 적용하고 운영할지를 정의하는 상위 전략 문서다. 운영 세부 규칙은 [[AGENTS.md]]를, 기존 개념 설명은 [[wiki/concepts/llm-wiki.md|llm-wiki]]를 참조한다.

## 1. 핵심 철학 — 왜 RAG가 아니라 위키인가

- **RAG는 매 질문마다 지식을 처음부터 재발견한다.** 축적이 없다. 다섯 문서를 종합해야 하는 질문은 매번 조각을 다시 찾아 맞춰야 한다.
- **위키는 지식을 한 번 컴파일해 두고 계속 최신 상태로 유지한다.** 교차 참조가 이미 거기 있고, 모순은 이미 표시되어 있으며, 종합은 지금까지 읽은 모든 것을 반영한다.
- **위키는 지속적이고 복리로 불어나는 산출물(compounding artifact)이다.** 소스를 추가할 때마다, 질문을 던질 때마다 더 풍부해진다.

> Obsidian은 IDE, LLM(나, 에이전트)은 프로그래머, 위키는 코드베이스다.

## 2. 역할 분담 (가장 중요)

| 사용자(human)가 할 일 | 에이전트(LLM)가 할 일 |
|---|---|
| 소스를 큐레이션하고 가져오기 | 읽고 요약하기 |
| 탐색 방향 제시 | 교차 참조 연결 |
| 좋은 질문 던지기 | 파일 정리·분류 |
| 의미 해석·판단 | 일관성 유지, 색인·로그 갱신 (잡일 전부) |

사용자는 **거의 위키를 직접 쓰지 않는다.** 소싱·탐색·질문이 사용자 몫이고, 나머지 모든 사무 작업(bookkeeping)은 에이전트가 한다. 위키가 유지되는 이유는 유지 비용이 0에 가깝기 때문이다 — LLM은 지루해하지 않고, 교차 참조 갱신을 잊지 않으며, 한 번에 15개 파일을 손볼 수 있다.

## 3. 3계층 아키텍처

1. **Raw 소스 (`raw/`)** — 불변(immutable). 진실의 원천. 에이전트는 읽기만 하고 절대 수정하지 않는다.
2. **위키 (`wiki/`)** — 에이전트 소유의 마크다운 페이지(요약·엔티티·개념·비교·질의응답). 에이전트가 생성·갱신·교차참조한다.
3. **스키마 (`AGENTS.md`)** — 구조·규약·워크플로우 정의. 에이전트를 "잡담 챗봇"이 아니라 "규율 있는 위키 관리자"로 만드는 설정 파일. 사용자와 함께 시간을 두고 공진화시킨다.

## 4. 세 가지 핵심 작업 (Operations)

### Ingest (수집)
사용자가 소스를 주면: ① `raw/`에 원본 저장 → ② 핵심 takeaway 논의 → ③ 기존 페이지 존재 여부 검색 → ④ 요약·엔티티·개념 페이지 작성/갱신(2개 이상 wikilink 필수) → ⑤ `index.md` 갱신 → ⑥ `log.md` 추가 → ⑦ 변경된 파일 보고. 한 소스가 5~15개 페이지를 건드리는 것이 정상이며 바람직하다(복리 효과).

### Query (질의)
사용자가 질문하면: ① `index.md` 먼저 읽기 → ② (100+ 페이지 시) `search_files`로 보강 → ③ 관련 페이지 읽기 → ④ 위키 기반으로 인용과 함께 종합 → ⑤ **재현하기 아까운 고가치 답변은 `queries/`에 페이지로 보존** → ⑥ 로그 기록. 탐색이 채팅 히스토리로 사라지지 않고 위키에 복리로 쌓이게 한다.

### Lint (점검)
주기적 건강검진: 고아 페이지, 깨진 wikilink, 색인 누락, 프론트매터 누락, 오래된(stale) 주장, 모순, 과대 페이지(200줄+), 태그 규약 위반, 로그 회전 필요 여부를 점검하고 심각도별로 보고한다.

## 5. 운영 규약 (Non-Negotiable)

- `raw/`는 절대 수정하지 않는다 — 수정은 위키 페이지에서만.
- **새 세션마다 먼저 오리엔트한다**: `AGENTS.md`(스키마) → `index.md` → 최근 `log.md` 순으로 읽고 시작. 건너뛰면 중복·누락이 발생한다.
- 모든 위키 페이지는 YAML 프론트매터를 가진다.
- 모든 새 페이지는 최소 2개의 다른 페이지와 연결한다.
- 모든 새 페이지는 `index.md`에 등록한다.
- 모든 작업은 `log.md`에 추가한다.
- 지나가는 언급(passing mention)에는 페이지를 만들지 않는다 (2개 이상 소스 또는 핵심성 기준).
- 모순은 조용히 덮어쓰지 않는다 — 양쪽 주장을 날짜·소스와 함께 기록하고 사용자 검토용으로 플래그한다.
- 한 번의 ingest가 10개 이상 기존 페이지를 갱신해야 하면 진행 전 사용자에게 확인한다.

## 6. 우리 위키의 현재 상태 및 개선 방향 (2026-06-10)

- 도메인: **AI 비즈니스 / AI 에이전트 / LLM 도구·워크플로우 / 교육 패러다임 변화 / 운영(operations)**. 한국어·영어 혼재.
- 구조 정리 필요 항목(향후 lint 시 처리):
  - `wiki/wiki/...` 중첩 경로와 루트 직속 한글 파일(`블로그 글 다듬기 가이드.md` 등)의 위치 일관성.
  - `summaries/`가 루트와 `wiki/` 양쪽에 분산 → 한 곳으로 통합 검토.
  - `index.md`가 일부 섹션만 채워져 있음 → 전체 페이지를 타입별로 재색인 검토.
- 다음 단계 제안: 전체 lint 1회 실행으로 색인 완전성·고아 페이지·중복 경로를 정리한 뒤, 이후부터 본 전략대로 ingest/query/lint 사이클을 돈다.
